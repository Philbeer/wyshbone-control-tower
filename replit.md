# Wyshbone Status Dashboard

## Overview
This project is a Node/Express application for monitoring Wyshbone applications. It provides a real-time, server-side rendered dashboard, a JSON API, and a file proxy for authenticated access. Its core purpose is to offer live insights into application health, performance, and code quality, track changes over time, and automate the evaluation of Wyshbone applications through an advanced suite of automated testing, diagnosis, and patch management tools. The project aims for comprehensive observability and automated evaluation.

## System Architecture
The application is built on Node.js and Express, utilizing server-side rendering with template literals and an in-memory Map for historical data. It polls Wyshbone app endpoints every 2 minutes, tracks the last 50 snapshots per source, computes metric deltas, and provides an auto-refreshing HTML dashboard. A robust evaluation suite automates testing, diagnosis, and patch management:

*   **Automated Investigations:** Manages triggers and diagnostic results, leveraging OpenAI GPT-4o-mini for diagnosis and patch suggestions.
*   **Automated Behaviour Tests:** Runs scenario-specific tests against Wyshbone UI endpoints, records, and displays results.
*   **Automated Detection & Investigation Triggering:** Initiates investigations for failures, timeouts, errors, and regressions.
*   **Patch Quality & Regression Protection:** Evaluates proposed patches in a sandbox to prevent regressions.
*   **Auto-Patch Generation:** LLM-powered patch generation using GPT-4o-mini, with automated evaluation.
*   **Behaviour Test Integration:** Integrates behaviour tests with the investigation system.
*   **Live User Run Logging & Investigation:** Logs real Wyshbone UI user conversations and enables investigation creation.
*   **Conversation Quality Investigation:** Analyzes and detects Wyshbone-specific conversation quality issues using GPT-4o-mini.
*   **Patch Failure Post-Mortem:** Analyzes rejected patches to classify failure reasons.
*   **Tower Verdict (Agent Loop Judgement):** Provides evidence-based constraint evaluation of `leads_list` artifacts, returning structured verdicts (ACCEPT/CHANGE_PLAN/STOP) with typed constraints (NAME_CONTAINS, NAME_STARTS_WITH, LOCATION, COUNT_MIN) and configurable `hardness` (hard/soft). It includes logic for `requested_count_user`, `attempt_history` for no-progress loops, and backwards compatibility for legacy constraints.
*   **Plastics Injection Moulding Rubric:** Judges `factory_state` and `factory_decision` artefacts based on criteria like `max_scrap_percent`, `scrap_rate_now`, and `energy_kwh_per_good_part` trends to determine verdicts (STOP, ACCEPT/CONTINUE, CHANGE_PLAN).
*   **ASK_LEAD_QUESTION Overconfidence Guard:** For `step_type=ASK_LEAD_QUESTION`, checks `metrics.confidence` and `metrics.evidence_items`. High confidence requires official site evidence or multiple independent domains; otherwise, it triggers CHANGE_PLAN with `overconfident_without_support` and `suggested_changes`. Handles hard and soft attribute verification, stopping for unverifiable hard constraints.
*   **Run Receipt Truth Judge:** Judges `run_receipt` artefacts for truth and consistency. Verifies: (A) `delivered_count` matches `delivered_leads.length` and counts are finite, (B) `contacts_proven=false` means counts are null and narrative doesn't claim absence, (C) contact counts match Tower-recomputed values from sibling `lead_pack`/`contact_extract` artefacts using proven extraction paths, (D) if matching to delivered leads is unreliable then `contacts_proven` must be false, (E) coverage counts are sane. Returns ACCEPT/RETRY/STOP with `ReceiptTruthMetrics` including `tower_computed_emails`, `tower_computed_phones`, `matching_reliable`, and per-rule results. Core logic: `src/evaluator/receiptTruthJudge.ts`. Tests: `tests/receiptTruthJudge.test.ts`. Route: handled in `server/routes-judge-artefact.ts` under `artefactType === "run_receipt"`.
*   **Artefact Judgement API:** Judges artifacts stored in Supabase by inspecting `payload_json` and applying rules based on `artefactType` (e.g., `leads_list`, `factory_state`/`factory_decision`, `run_receipt`), returning both legacy `verdict` and canonical `towerVerdict`.
*   **Tower Verdict Persistence:** All verdict decisions are persisted to the `tower_verdicts` Supabase table.
*   **Structured Stop Reasons:** Every STOP and CHANGE_PLAN verdict includes a structured `stop_reason` with a stable `code`, `message`, and optional `evidence`.
*   **Proof Mode:** Supports `proof_mode` for deterministic demo behavior for both `/tower-verdict` and `/judge-artefact` routes, with proof verdicts also persisted.
*   **Evidence Quality Judge:** A post-check module enforcing evidence discipline on web-derived claims, triggering STOP for issues like `VERIFIED_WITHOUT_EVIDENCE`, `VERIFIED_EXACT_BELOW_REQUESTED`, `DELIVERY_SUMMARY_MISMATCH`, `NO_EVIDENCE_PRESENT`, or `PASS_WITHOUT_VERIFICATION`. The evidence quality override only converts ACCEPTâ†’STOP (never overrides CHANGE_PLAN to STOP). For CHANGE_PLAN, gaps are appended but the verdict is preserved. **Self-evident bypass (TOWER_SELF_EVIDENT_FIX):** When all hard constraints are self-evident types (NAME_CONTAINS, NAME_STARTS_WITH, LOCATION, COUNT_MIN), the ACCEPTâ†’STOP override is skipped â€” these constraints don't require external evidence fields on leads. **Discovery-only bypass:** When `verification_policy=DIRECTORY_VERIFIED` and `strategy=discovery_only`, the evidence quality override is also skipped â€” these runs never attempted website evidence collection. **Count bug fix:** When no user-requested count exists, `requested_count` is passed as `null` to the evidence quality judge, preventing false `VERIFIED_EXACT_BELOW_REQUESTED` failures from the candidate pool size (default 20).
*   **Behaviour Judge:** A fire-and-forget LLM judge (gpt-4o, configurable via `BEHAVIOUR_JUDGE_MODEL`) that classifies every leads_list verdict into one of five outcomes: PASS, HONEST_PARTIAL, BATCH_EXHAUSTED, CAPABILITY_FAIL, WRONG_DECISION. Runs asynchronously after Tower produces its verdict â€” does not add latency to the verdict response. Gated behind `BEHAVIOUR_JUDGE_ENABLED=true`. Results persisted to `behaviour_judge_results` table. Input: original goal, strategy, verification policy, delivered/requested counts, constraint verdicts, Tower verdict/gaps, and `agent_clarified` flag. Core logic: `src/evaluator/behaviourJudge.ts`. Fired from `server/routes-tower-verdict.ts`.
*   **Tower Dev Chat:** An interface for developers to report issues, with automatic context gathering and AI-powered patch suggestions using GPT-4o-mini, tracked in PostgreSQL.
*   **Learning Layer v1 â€” Tower Policy Updates:** Tower judges whether to update policies (`radius_policy_v1`, `enrichment_policy_v1`, `stop_policy_v1`) based on evidence (`decision_log`, `outcome_log`, `telemetry`, `current_policy`). It applies a rubric with checks for evidence completeness, sample size, regression detection, success rate, magnitude guards, and confidence threshold.
*   **Learning Layer v1 â€” max_replans Learning (`POST /api/tower/learn-max-replans`):** Updates `stop_policy_v1.max_replans` per `scope_key` based on last N run outcomes. Computes `replan_helped_rate`, `waste_rate`, and `exceeded_rate`. Update logic: if exceeded>=0.30 AND helped>=0.50 â†’ +1 (cap 3); if waste>=0.60 â†’ -1 (floor 0); else no_learn. Guardrails: N>=5 required, max change Â±1, blocks update if last 3 runs all FAIL. Persists to `policy_versions` and emits `policy_update` or `no_learn` artefacts with evidence summaries and rollback pointers. Core logic: `src/evaluator/maxReplansLearning.ts`. Tests: `tests/maxReplansLearning.test.ts`.
*   **Learning Update Emitter (Tower-driven policy signals):** Tower evaluates every leads_list verdict for learning triggers and emits a `learning_update` artefact when actionable. Core logic: `src/evaluator/learningUpdateEmitter.ts`. Wired into both `/api/tower/tower-verdict` and `/api/tower/judge-artefact` routes. Emission rules: (1) CHANGE_PLAN with `underfilled_results` (delivered < requested) increases `search_budget_pages` by +1 (cap 3). (2) CHANGE_PLAN with `weak_verification` (hard constraint unknown/unverified) escalates `verification_level` one step (minimal->standard->strict). (3) ACCEPT with `too_slow` (replans>=2 or steps>15) decreases `verification_level` one step or `search_budget_pages` by -1 (floor 1). (4) STOP verdicts emit no learning update. (5) Normal ACCEPT with no trigger emits nothing. Metrics produced: `requested_count`, `returned_count`, `underfilled` (bool), `verification_quality_score` (weak/ok/strong), `cost_or_time_signal` (steps_count, tool_calls, duration_bucket, replans_used), `failure_reason` enum. Artefact schema: `type: "learning_update"`, `query_shape_key`, `changed_fields: [{field, before, after}]`, `tower_reason`, `metrics_trigger`, `run_id`. Tower does NOT write to LearningStore directly; Supervisor reads the artefact and persists it. Request fields: `query_shape_key` (required for emission), `steps_count`, `tool_calls`, `current_search_budget_pages`, `current_verification_level`, `current_radius_escalation`.

*   **Concatenation Artifact Detection:** A lightweight sanity check in `judgeLeadsListCore` (runs after NO_PROGRESS, before constraint evaluation) that flags corrupted inputs. Detects: (1) full questions embedded in parentheses, (2) suspicious mid-word capitalisation, (3) triple-repeated words, (4) verb-word concatenation without spaces (e.g., "Walescan" â†’ "Wales" + "can"). Uses a safe-word list to avoid false positives on legitimate words (American, candidate, tuscan, Vatican, etc.). Returns `CHANGE_PLAN` with code `INPUT_CONCATENATED` and suggests the user restate the request. Core function: `detectConcatenationArtifacts()` in `towerVerdict.ts`.

**Honest Partial Success (HONEST_SHORTFALL):**
When a run delivers verified leads but falls short of the requested count, Tower now detects honest partial success rather than treating it as a generic failure. Detection criteria: (1) `deliveredCount > 0`, (2) `delivery_summary` is `"PARTIAL"` or `"PASS"`, (3) `verified_exact_count` from verification_summary (or computed from lead-level evidence) is > 0 but < requested, (4) no non-COUNT_MIN hard violations exist, (5) no non-HAS_ATTRIBUTE hard unknowns exist. When detected: if replans exhausted â†’ `ACCEPT_WITH_UNVERIFIED` with gap `HONEST_SHORTFALL` and `stop_reason.code = "HONEST_SHORTFALL"`; if replans available â†’ `CHANGE_PLAN` with gap `HONEST_SHORTFALL`. The `delivered` count in the verdict reflects `verifiedExactCount` (not raw lead count), and `stop_reason.evidence` includes `verified_exact_count`, `shortfall`, `delivery_summary`, and `hard_evidence_constraints_passed`. This path fires before both the generic hard-violations exit (when only COUNT_MIN is violated) and the count-shortfall branch. Core function: `detectHonestPartial()` in `towerVerdict.ts`. Tests: 6 honest shortfall tests in `tests/towerVerdict.test.ts`.

**Constraint-Driven Evidence-Based Evaluation (v3 + Phase 4 claim-sensitive judgement):**
Tower uses user intent (`requested_count_user`) and accumulated matching results (`delivered_matching_accumulated`). It defines a priority for resolving requested and delivered counts. Constraints are typed (NAME_CONTAINS, NAME_STARTS_WITH, LOCATION, COUNT_MIN, HAS_ATTRIBUTE) with `hardness` (hard/soft), and a clear resolution priority for different constraint formats. Logic includes replan-aware verdicting, typed `suggested_change` types, and detection of `label_misleading` gaps. Hard constraints are strictly enforced, never auto-relaxed. CVL-aware judgement uses `verification_summary` for evidence-based verification, prioritizing `verified_exact_count` and requiring CVL status for LOCATION constraints. Rationale wording is factual and aligned with `delivery_summary`.

**PHASE_4 â€” Claim-Sensitive Judgement:**
Per-constraint judgement now considers evidence requirements, proof burden, source tier, and negation:
- **Types:** `EvidenceRequirement` (none/lead_field/directory_data/search_snippet/website_text/external_source), `ProofBurden` (self_evident/evidence_required/evidence_required_first_party/inherently_uncertain), `SourceTier` (first_party_website/directory_field/search_snippet/lead_field/external_source/unknown), `ConstraintVerdict` (VERIFIED/PLAUSIBLE/UNSUPPORTED/CONTRADICTED/NOT_APPLICABLE). All exported from `towerVerdict.ts`.
- **Constraint interface** extended with optional `evidence_requirement` and `label`. `ConstraintResult` extended with optional `constraint_verdict`. `AttributeEvidenceArtefact` extended with optional `source_tier`. `StructuredConstraint` extended with `evidence_requirement` and `label`.
- **Proof burden mapping:** `proofBurdenFromRequirement()` maps evidence_requirement â†’ ProofBurden. none/lead_field â†’ self_evident; directory_data/search_snippet â†’ evidence_required; website_text â†’ evidence_required_first_party; external_source â†’ inherently_uncertain; default â†’ evidence_required.
- **Source tier confidence capping:** `applyProofBurdenCap()` in `semanticEvidenceJudge.ts` â€” for constraints with proof burden `evidence_required_first_party`, non-first-party evidence cannot reach `verified` status (capped to `weak_match` with confidence â‰¤0.6).
- **Constraint verdict derivation:** `deriveConstraintVerdict()` maps (passed, status, semanticStatus, proofBurden, sourceTier) â†’ ConstraintVerdict. `not_applicable` â†’ NOT_APPLICABLE; `contradicted` â†’ CONTRADICTED; passed+verified+first_party â†’ VERIFIED; passed+verified+non-first-party for website_text burden â†’ PLAUSIBLE; weak_match â†’ PLAUSIBLE; self_evident+passed â†’ VERIFIED. Backward compatible: VERIFIED/PLAUSIBLE â†’ `passed: true`, UNSUPPORTED/CONTRADICTED â†’ `passed: false`.
- **NAME_CONTAINS/NAME_STARTS_WITH:** Now fall through to local string verification when CVL returns `unknown` or `not_attempted` (self-evident claims no longer blocked by CVL). Local verification returns VERIFIED/UNSUPPORTED.
- **HAS_ATTRIBUTE:** Tracks per-lead verdicts using source_tier from evidence artefacts. Aggregate constraint_verdict is the best lead verdict (VERIFIED > PLAUSIBLE) or CONTRADICTED if any evidence contradicts and no positive evidence exists.
- **Negation in keyword fallback:** Detects negation patterns within 50-char window around matches. All-negated matches return `contradicted` status with `satisfies: "no"`.
- **Semantic-verify endpoint:** Accepts optional `source_tier` and `evidence_requirement` fields to enable proof burden capping on standalone calls.
- **Run verdict derivation uses constraint_verdict** (Phase 5). See PHASE_5 section below.

**PHASE_5 â€” Constraint-Verdict-Driven Run Verdict:**
Run verdict in `judgeLeadsListCore` now derives from `constraint_verdict` on each hard constraint result, not just `passed: boolean`. Rules:
- **All hard VERIFIED â†’ ACCEPT** (unchanged behavior for fully verified constraints).
- **All hard VERIFIED or PLAUSIBLE (none worse) â†’ ACCEPT_WITH_UNVERIFIED** (new: PLAUSIBLE constraints no longer silently pass as ACCEPT). LOCATION constraints excluded from PLAUSIBLE downgrade for backward compat â€” LOCATION without CVL is structurally plausible and always treated as passing.
- **Any hard CONTRADICTED â†’ immediate STOP** with code `HARD_CONSTRAINT_CONTRADICTED`, no replan offered. Fires before all other verdict paths.
- **Any hard UNSUPPORTED + replans â†’ CHANGE_PLAN** (existing behavior).
- **Any hard UNSUPPORTED + no replans â†’ STOP** (existing behavior).
- **Count not met + no hard violations â†’ CHANGE_PLAN/STOP** (unchanged).
- **Response shape additions** (`TowerVerdict` interface):
  - `failing_constraint_id?: string` â€” ID of the worst-performing hard constraint.
  - `failing_constraint_reason?: string` â€” human-readable reason for the failure.
  - `hard_constraint_verdicts?: HardConstraintVerdictEntry[]` â€” array of `{ id, verdict, label }` for every hard constraint.
- **New type:** `HardConstraintVerdictEntry` exported from `towerVerdict.ts`.
- **Exported:** `CONSTRAINT_VERDICT_RANK` for external ranking use.
- **Helper functions:** `buildHardConstraintVerdicts()`, `findFailingHardConstraint()`, `attachPhase5Fields()` â€” all in `towerVerdict.ts`.
- **COUNT_MIN** now carries `constraint_verdict` (VERIFIED if passed, UNSUPPORTED if not).
- **Supervisor handoff:** Phase 5 fields flow through `res.json({ ...result })` spread in routes â€” no route changes needed. Supervisor reads `result.verdict`, `result.action`, `result.constraint_results`, `result.stop_reason`, and now also `result.failing_constraint_id`, `result.failing_constraint_reason`, `result.hard_constraint_verdicts`.

**Delivered Count Resolution (v2):**
`resolveDeliveredCount()` returns `{ count, source }` for debug tracing. Priority chain (TOWER_COUNT_FIX applied): (1) `delivered_leads.length`, (2) `delivered_count` (**moved up** â€” authoritative delivery signal from Supervisor, prevents search pool contamination), (3) `leads.length`, (4) CVL `verification_summary.verified_exact_count`, (5) top-level `verified_exact`, (6) `accumulated_count`, (7) `delivered.delivered_matching_accumulated`, (8) `delivered.delivered_matching_this_plan`, (9) numeric `delivered`, (10) `matchedLeadCount`, (11) `default(0)`. Fields `verified_exact` and `delivered_leads` added to `TowerVerdictInput` and Zod schema.

**Verdict Debug Block:**
Every `TowerVerdict` includes `_debug: { extractedDeliveredCount, extractedRequestedCount, source }` showing which field was used to compute `deliveredCount`, for traceability in artefacts and API responses.

**Null Requested Count (No Count Requirement):**
When `requested_count_user` is null/missing (user did not request a specific number), Tower no longer fails with `MISSING_REQUESTED_COUNT`. Instead: (1) If `delivered_exact_count >= 1` and all hard constraints are satisfied â†’ ACCEPT with `requested: 0`. (2) If `delivered_exact_count >= 1` but hard constraints are violated â†’ STOP with `COUNT_MET_HARD_VIOLATED`. (3) If `delivered_exact_count >= 1` but hard constraints are unknown and replans available â†’ CHANGE_PLAN. (4) If `delivered_exact_count >= 1` but hard constraints unknown and no replans â†’ ACCEPT_WITH_UNVERIFIED. (5) If `delivered_exact_count === 0` â†’ STOP with code `ZERO_DELIVERED`. Count shortfall logic (`INSUFFICIENT_COUNT`) only applies when the user explicitly requested a numeric count.

**Contract Validation:**
`judgeLeadsListCore` performs an early contract check: if no leads/delivered_leads array AND no reliable delivered-count fields are present, it returns STOP with code `CONTRACT_ERROR`. The rationale and stop_reason.message start with "Contract error:" and list all missing fields by name. "No results were found" is never used for contract errors.

**Route Wiring (fixed in both routes):**
Both `judgeLeadsListArtefact` (routes-judge-artefact.ts) and the tower-verdict inline route (routes-tower-verdict.ts) now pass `delivered_count`, `accumulated_count`, `verified_exact`, and `delivered_leads` through to `judgeLeadsList()`. The `deliveredObj` IIFE in judge-artefact now handles numeric `delivered` at the payload root. Both routes log the exact payload received (`[TOWER_IN] final_delivery_payload:`) and the verdict output includes `_debug` in logs. The `_debug` block is forwarded through `metrics._debug` in judge-artefact responses.

**resolveLeads() fallback to delivered_leads:**
`resolveLeads()` now checks `delivered_leads` when `leads` array is empty/missing. This ensures the evidence quality judge receives lead data even when only `delivered_leads` is present in the payload, preventing false ACCEPTâ†’STOP overrides due to "NO_EVIDENCE_PRESENT".

**Type-safe count resolution (resolveRequestedCount / route extraction):**
`resolveRequestedCount` now uses `coerceToNumber()` to validate every candidate is a finite number. Non-numeric strings like `"explicit"` are skipped, falling through to the next candidate (e.g. `requested_count: 10`). `judgeLeadsListArtefact` in routes-judge-artefact.ts uses the same `toFiniteNumber()` guard when extracting `requestedCountUser`, `requestedCount`, and `targetCount` from `successCriteria` and `payloadJson`. Utility `isFiniteNumber()` is used throughout towerVerdict.ts for type guards.

**"Unexpected state" fallback safety:**
The INTERNAL_ERROR fallback at the end of `judgeLeadsListCore` coerces both `deliveredCount` and `requestedCount` to safe numerics via `isFiniteNumber()`, then checks `numericDelivered >= numericRequested && hardViolations === 0 && hardUnknowns === 0` â€” if met, returns ACCEPT. Logs all computed variables (with types) before the guard. The STOP fallback still exists for genuine unhandled states with full `_debug` tracing.

**Count Met + Hard Violated:**
When `deliveredCount >= requestedCount` but hard constraint violations exist, Tower returns STOP with code `COUNT_MET_HARD_VIOLATED` listing the violated fields. Previously this was an empty if block that fell through to incorrect logic paths.

**HAS_ATTRIBUTE Constraint & attribute_evidence (CVL):**
CVL evaluation for `HAS_ATTRIBUTE` constraints (e.g., `c_attr_live_music`) consumes `attribute_evidence` artefacts from Supervisor. When evaluating a lead for HAS_ATTRIBUTE: if an `attribute_evidence` artefact exists for that lead+attribute, the constraint status is set to the artefact's verdict (yes/no/unknown) with confidence and evidence pointers (evidence_id, source_url, quote). If no artefact exists, status defaults to `unknown` with low confidence. Unknown is never treated as false â€” it excludes the constraint from hard violations and prevents `verified_exact` from being true until all hard constraints are resolved to `yes`. The `ConstraintResult` exposes `status`, `evidence_id`, `source_url`, `quote`, and `attribute_evidence_details` for UI consumption. The `routes-judge-artefact.ts` fetches `attribute_evidence` artefacts from the `artefacts` table (type=`attribute_evidence`) for the same `run_id` and passes them into `judgeLeadsListArtefact`.

**Semantic Evidence Judge (`src/evaluator/semanticEvidenceJudge.ts`):**
Tower now performs LLM-based semantic evaluation of attribute evidence before the deterministic verdict pipeline runs. When `attribute_evidence` artefacts exist with evidence text and the constraints include `HAS_ATTRIBUTE`, both route handlers (`routes-judge-artefact.ts` and `routes-tower-verdict.ts`) call `enrichAttributeEvidence()` which runs LLM judgement (GPT-4o-mini by default, configurable via `SEMANTIC_JUDGE_MODEL` env var) on each evidence item. The semantic judge acts as Tower's judgement layer: it judges whether evidence *produced by a tool* actually supports the user's real constraint â€” not just whether the tool ran successfully. A successful crawl is NOT a successful verification. Evidence input priority: `extracted_quotes` (array of snippets from page text) is used as primary evidence; falls back to single `quote` field if no extracted quotes exist. Additional context passed to the judge: `page_title`, `constraint_raw`, `attribute_raw`, `source_url`. LLM output format: `judgement_type` ("attribute_verification"), `satisfies` (boolean), `status` (verified/weak_match/no_evidence/insufficient_evidence), `strength` (strong/indirect/weak/none), `confidence` (0-1), `reason` (string), `supporting_quotes` (up to 3 short quotes extracted from evidence, never invented). Mapping to verdict: boolean `satisfies` â†’ "yes"/"no"; string "yes"/"no"/"unknown" also accepted; status-only fallback: verifiedâ†’"yes", no_evidenceâ†’"no", weak_match/insufficient_evidenceâ†’"unknown". "direct" strength is auto-mapped to "strong" for backward compat. Fields on `AttributeEvidenceArtefact`: `extracted_quotes` (string[]), `page_title`, `constraint_raw`, `attribute_raw`, `semantic_verdict` (yes/no/unknown), `semantic_status` (verified/weak_match/no_evidence/insufficient_evidence), `semantic_strength` (strong/indirect/weak/none), `semantic_confidence` (0-1), `semantic_reasoning`, `semantic_supporting_quotes` (string[]). In `towerVerdict.ts`, the `evaluateConstraint` HAS_ATTRIBUTE path prefers `semantic_verdict` over the upstream `verdict` when present (`effectiveVerdict = ev.semantic_verdict ?? ev.verdict`). Fallback: if the LLM call fails or OPENAI_API_KEY is not set, the original upstream verdict is preserved and used â€” no behaviour change. Set `DEBUG_TOWER_SEMANTIC_TRACE=true` to enable detailed semantic judgement logging. The semantic enrichment runs in parallel via `Promise.allSettled` for all evidence items. Core functions: `judgeEvidenceSemantically()` (single item, accepts extractedQuotes/pageTitle/constraintRaw/attributeRaw), `enrichAttributeEvidence()` (batch pre-processing).

**Semantic Verify Endpoint (`POST /api/tower/semantic-verify`):**
Standalone endpoint for Supervisor to request semantic evidence judgement without going through the full judge-artefact or tower-verdict pipelines. Accepts: `original_user_goal` (required), `constraint_to_check` (required), `lead_name` (required), `source_url`, `evidence_text` (string or array), `extracted_quotes` (string array), `page_title`, `attribute_raw`, `constraint_raw`. Always returns JSON (never HTML). Response shape: `{ judgement_type, satisfies (boolean), status, strength, confidence, reason, supporting_quotes, matched_snippets, judge_mode }`. `matched_snippets` is an alias for `supporting_quotes` (for Supervisor compatibility). `judge_mode` is either `"llm"` (when full LLM judge ran) or `"keyword_fallback"` (when LLM was unavailable and deterministic keyword matching was used). Route file: `server/routes-semantic-verify.ts`, mounted at `/api/tower` in `server.js`.

**Verbatim Match Pre-Check:** Before the LLM or keyword fallback runs, `checkVerbatimMatch()` checks whether ANY evidence text contains the constraint value as a contiguous, multi-word phrase (case-insensitive). `normalizeConstraintForVerbatim()` strips `c_attr_` prefixes and converts underscores/hyphens to spaces, so `"live_music"` â†’ `"live music"`. Single-word constraints are rejected (requires at least one space). If found, returns `verified` with `strength: "strong"`, `confidence: 0.95`, and up to 3 context-windowed snippets. Word-boundary and negation checks still apply. Example: constraint `"live_music"` and quote `"From live music and quiz nights..."` â†’ `verified` immediately; quote `"watch every six nations game live at the brewery"` â†’ no match (only the word "live", not the full phrase "live music") â†’ falls through to keyword fallback â†’ `weak_match`. Judge mode is `"keyword_fallback"`.

**Keyword Fallback Judge:** When the LLM is unavailable (no API key, LLM call fails, or response unparseable), the semantic judge falls back to a deterministic keyword-based judge (`keywordFallbackJudge()` in `semanticEvidenceJudge.ts`). It tokenizes the constraint value and all evidence texts (including page title), checks for keyword overlap, and returns: `weak_match` with supporting quotes if â‰Ą50% token overlap found, `no_evidence` if no meaningful overlap. **PHASE_4: negation detection** â€” before counting a match as positive, checks for negation patterns (no, not, never, without, stopped, no longer, etc.) within a 50-character window around the match. If ALL matches are negated, returns `contradicted` status with `satisfies: "no"` instead of `weak_match`. The keyword judge is conservative: it requires real textual overlap, caps at 3 quotes, and clearly labels its output with `judge_mode: "keyword_fallback"`. The LLM judge is always preferred when available.

**HAS_ATTRIBUTE diagnostic trace:** Set env `DEBUG_TOWER_ATTR_TRACE=true` to enable detailed decision tracing for HAS_ATTRIBUTE constraints. Trace logs (prefixed `[TOWER][ATTR_TRACE]`) cover: constraint input, cvlMatch lookup, per-lead evidence matching, verdict-level summary (hardUnknowns/hardViolations counts), and STOP/CHANGE_PLAN path decision (anyUnverifiable, canReplan). Trace does not change behavior.

**HAS_ATTRIBUTE verdict paths (when count is met but hard unknowns exist):**
1. If CVL reason contains "unverifiable" AND can't replan â†’ STOP with code HARD_CONSTRAINT_UNVERIFIABLE
2. If can replan â†’ CHANGE_PLAN
3. If can't replan and no unverifiable flag â†’ STOP with code HARD_CONSTRAINT_UNKNOWN (hard_unknown_no_replans)
Key: if no `attribute_evidence` artefacts exist in DB AND no CVL constraint_results entry matches, the constraint evaluates to status=unknown, goes into hardUnknowns, and triggers path 2 or 3.

**Relationship Predicate Enforcement:**
Tower enforces relationship evidence gating as a post-check in `judgeLeadsList`. If `requires_relationship_evidence === true` AND `verified_relationship_count === 0` (or undefined), ACCEPT/PASS is illegal. Tower overrides to STOP with three distinct codes: `RELATIONSHIP_VERIFICATION_NOT_ATTEMPTED` (field absent entirely), `RELATIONSHIP_EVIDENCE_MISSING` (field present but zero verified), `RELATIONSHIP_CHECKED_NO_MATCH` (checked but no match found). Both routes (`/tower-verdict` and `/judge-artefact`) pass these fields through from the request/payload. The gate runs after evidence quality checks but before final return, ensuring it is the last honesty check. Input fields: `requires_relationship_evidence` (boolean), `verified_relationship_count` (number) â€” both optional, backward compatible.

**CvlConstraintStatus:** `"yes" | "no" | "unknown" | "not_attempted" | "not_applicable"`. HAS_ATTRIBUTE fallback uses `"not_attempted"` (not `"unknown"`) when no evidence exists, making the distinction between "checked but inconclusive" vs "never checked" explicit. Both `"unknown"` and `"not_attempted"` are treated as hard unknowns in CVL constraint evaluation. `"not_applicable"` is treated as a VALID state (e.g. location irrelevant for online businesses): it passes the LOCATION constraint check, does not trigger the truth gate, and is not counted as missing evidence. Zod schema accepts all five statuses.

**Artefact Type Equivalence:** `final_delivery` is accepted as equivalent to `leads_list` in both `/api/tower/tower-verdict` and `/api/tower/judge-artefact` routes. Both types flow through the same leads_list judging logic.

**Verification-Attempted Rubric (documented in `src/evaluator/evidenceQualityJudge.ts`):** Tower considers "verification attempted" when at least one of these fields is present on any lead: `verified` (boolean), `evidence` (string/array/object), `source_url` (string). If none exist on any lead AND `verified_exact_count` is missing, verdict is STOP with gap `NO_EVIDENCE_PRESENT`. Per-lead: verified=true + evidence/source_url = verified_with_evidence; verified=true + no evidence = verified_without_evidence (gap); verified=false = counted, not penalised; verified=undefined = unknown_count (not penalised).

**API Schema (delivery_summary):** The `/api/tower/tower-verdict` route validates `delivery_summary` as `z.enum(["PASS", "PARTIAL", "STOP"])`, rejecting arbitrary string values. The learning layer route uses `["PASS", "PARTIAL", "STOP", "FAIL"]`.

**Verdict Types & UI Mapping:**
`TowerVerdictAction = "ACCEPT" | "ACCEPT_WITH_UNVERIFIED" | "CHANGE_PLAN" | "STOP"`. The `VERDICT_UI_MAP` export provides UI rendering guidance: ACCEPT â†’ "Verified satisfied" (success), ACCEPT_WITH_UNVERIFIED â†’ "Ran, but not verified â€” best-effort accepted" (warning), CHANGE_PLAN â†’ "Replanning" (warning), STOP â†’ "Cannot meet requirements honestly" (error). `ACCEPT_WITH_UNVERIFIED` maps to `action: "continue"` â€” execution proceeds but the UI must show a warning badge. Plain ACCEPT means all hard constraints are verified to the declared standard. Input field: `best_effort_accepted?: boolean` â€” when true and unresolved hard constraints exist, verdict becomes ACCEPT_WITH_UNVERIFIED instead of STOP. Without `best_effort_accepted`, any unresolved hard constraint forces STOP.

**must_be_certain Backstop:**
Tower enforces an absolute certainty backstop: if any `UnresolvedHardConstraint` has `must_be_certain=true` AND `verifiability` is `"proxy"` or `"unverifiable"`, verdict is forced to STOP with code `MUST_BE_CERTAIN_VIOLATED` â€” regardless of `best_effort_accepted`. This gate runs before the constraint gate and cannot be overridden. Stop reason message: `"User required certainty for "<label>", but evidence is not strictly verifiable (verifiability: <type>)."` Verifiable constraints with `must_be_certain=true` pass through normally (certainty is achievable). Tests: 5 must_be_certain tests in `tests/towerVerdict.test.ts`.

**Constraint Gate Check:**
Tower enforces a strict "Constraint Gate Check" in `judgeLeadsList` that prevents ACCEPT when any hard constraints declared by Supervisor are unresolved. Supervisor sends `unresolved_hard_constraints` (array of `{ constraint_id, label, verifiability, proxy_selected? }`). For each entry: if `verifiability="unverifiable"` â†’ blocked; if `verifiability="proxy"` and `proxy_selected` is null/empty â†’ blocked; if `verifiability="verifiable"` â†’ blocked. Without `best_effort_accepted`: blocked entries override ACCEPTâ†’STOP with code `CONSTRAINT_GATE_BLOCKED`. With `best_effort_accepted=true`: blocked entries override ACCEPTâ†’ACCEPT_WITH_UNVERIFIED with code `CONSTRAINT_GATE_BEST_EFFORT`. The gate runs before the time predicate gate and after the relationship gate.

**Truth Gate (Final Gate):**
After all other gates, a final truth gate checks `constraint_results` for any hard constraint with `passed=false`. If found and `best_effort_accepted=true` â†’ ACCEPT_WITH_UNVERIFIED (code: `TRUTH_GATE_BEST_EFFORT`). If found without best_effort â†’ STOP (code: `TRUTH_GATE_BLOCKED`). This catches constraint failures that bypass the explicit `unresolved_hard_constraints` path. Tests: 6 constraint gate + 6 truth/ACCEPT_WITH_UNVERIFIED tests in `tests/towerVerdict.test.ts`.

**Time Predicate Honesty Enforcement:**
Tower enforces time-based constraint honesty as a post-check in `judgeLeadsList`. **Tower never infers or reinterprets time predicates from goal text** â€” it reads only what Supervisor declared via input fields. Input fields: `time_predicates` (array of `{ predicate, hardness }`), `time_predicates_mode` (`"verifiable" | "proxy" | "unverifiable"`), `time_predicates_proxy_used` (nullable enum: `"news_mention" | "recent_reviews" | "new_listing" | "social_media_post" | "press_release"`), `time_predicates_satisfied_count`, `time_predicates_unknown_count`. If Supervisor sends `time_predicates` but omits `time_predicates_mode`, Tower treats all hard predicates as blocked (cannot assume satisfied). `detectTimePredicate()` is exported as a utility for Supervisor to call, but Tower's gate does not invoke it. Verdict rules: (1) If any hard time_predicate is blocked (unverifiable, proxy not run, or 0 satisfied), verdict overrides ACCEPTâ†’STOP with code `TIME_PREDICATE_BLOCKED`. (2) If proxy used with evidence but `delivery_summary` doesn't say "proxy" or "PARTIAL", verdict overrides to STOP with code `TIME_PREDICATE_PROXY_LANGUAGE_MISMATCH`. (3) Soft predicates do not block ACCEPT. Output includes `user_summary` one-liner for UI display; wording never says "verified" for proxy-based results. Stop codes: `TIME_PREDICATE_BLOCKED`, `TIME_PREDICATE_PROXY_LANGUAGE_MISMATCH`. Tests: 22 time predicate tests in `tests/towerVerdict.test.ts`.

**Relationship Predicate Auto-Detection:**
Tower auto-detects relationship predicates from the goal text via `detectRelationshipPredicate()`. Detected predicates: "works with", "work with", "working with", "supplies", "supply", "supplying", "serves", "serve", "serving", "supports", "support", "supporting", "partners with", "partner with", "partnering with", "provides services to", "provide services to", "providing services to", "contracted by", "contracted to". When detected, the relationship gate fires even without explicit `requires_relationship_evidence=true` from Supervisor. The `stop_reason.evidence` includes `detected_predicate` and `detection_source` for traceability. Results are treated as candidates only â€” PASS requires â‰Ą1 verified relationship match with website/case-study/third-party evidence.
