BUG FIX — Tower falsely rejecting deliveries due to unnecessary input corruption detection

PROBLEM: Tower is rejecting valid deliveries containing 11+ verified leads with:
"Input appears corrupted. Input appears concatenated: 'organisations' looks like words merged without spaces."

"Organisations" is a normal English word. Tower is hallucinating a corruption problem that doesn't exist. This has blocked results from displaying on EVERY run with this query.

ROOT CAUSE: Find where Tower checks for corrupted/concatenated input. This could be:
A) Instructions in the Tower verdict LLM system prompt telling it to check for corruption
B) A code-level regex/heuristic pre-check
C) Both

THE FIX:

If the check is in the LLM system prompt — REMOVE the instruction about checking for corrupted, concatenated, garbled, or malformed input text entirely. By the time a delivery reaches Tower for final judgement, the input has already passed through intent extraction, constraint mapping, planning, search execution, and evidence gathering. If the input was genuinely corrupted, those steps would have failed. Tower should not be doing input validation or spell-checking. Its job is to judge whether the delivery answers the question with sufficient evidence.

If the check is in code — remove it unless it has a clear documented purpose for catching genuinely garbled input (like "findorganisationsthatworkwith" with literally no spaces). If keeping it, make it only flag strings where there are NO spaces at all, not strings containing individual long words.

IMPORTANT:
- Do NOT add a whitelist of British English words — that's curve-fitting, not a fix
- REMOVE or drastically narrow the corruption check
- Do NOT modify any other Tower logic (per-candidate verification, evidence judging, GT lookup, BJ scoring)
- After making changes, RESTART the Tower server explicitly

After fixing, append findings to AGENT_LOG.md documenting: where the corruption check was, what it did, and what was removed/changed.
